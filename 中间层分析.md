## 0. 执行摘要(TL;DR)

| 维度   | 核心结论                                                                                                   |
| ---- | ------------------------------------------------------------------------------------------------------ |
| 协议   | **GraphQL-over-HTTP POST**(查询)+ **REST**(管理)。GraphQL 查询体被包装为 `{"apiQuery":"...","isTest":true}` 发往单一端点 |
| 鉴权   | **无 Bearer Token**。依赖宿主门户注入 `window.currentLoginUser` + 请求头 `x-operator-id`/`tenantId`/`appId`/`cftk`  |
| 查询方言 | **非标准 GraphQL**,通过 6 个自定义指令扩展:`@limit`/`@offset`/`@where`/`@order`/`@function`/`@connect`              |
| 元数据  | **指标目录/维度目录接口完全缺失**(硬依赖,需补建);表目录部分具备(`services/list` + `__type` 内省)                                    |
| 同环比  | **无内置**,通过 SQL `case when`/`coalesce` 在数据服务定义层手工计算                                                     |
| 分页   | `@limit`+`@offset` 模式,**响应不返回总条数**                                                                     |
| 联动   | 通过 `globalVars`/`localVars` + `updateGlobalVar()` 实现图表点击联动                                             |
| 权限   | 数据级行级过滤通过"控制字段(ControlField)"由后端下推执行,前端不做过滤                                                            |
| 微前端  | 通过 single-spa + qiankun 双兼容接入宿主门户                                                                      |

## 1. 表服务接口形态

### 1.1 协议形态

**GraphQL-over-HTTP POST + REST 管理端点混合架构**。

* **数据查询**:GraphQL 查询字符串被包装在 JSON 字段 `apiQuery` 中,通过 HTTP POST 发送到单一端点

  * 端点:`POST rest/cbc/cbcbidynamicapiservice/v1/graphql`
  * 请求体:`{"apiQuery": "{ ... }", "isTest": true}`
    — 来源:`website-src/src/app/component/http/GraphqlHttpManager.ts:9,23,28`;`website-src/src/app/component/editor/http/DataSourceHttpManager.ts:11,29-31`
* **批量查询**:同一端点支持 `?ids=ds1,ds2,...`,批量大小上限 **5**
  — 来源:`website-src/@cloud/datav-lowcode/datasource/GraphqlGroupEngine.ts:10,12,48-53`
* **REST 管理端点**:数据服务 CRUD、审批、元数据查询、卡片执行等走标准 REST,集中在以下前缀:

  * `rest/cbc/cbcbidataservice/v1/` — 数据服务管理(卡片、服务目录、审批、收藏、元数据、标签中心)
  * `rest/cbc/cbcbidynamicapiservice/v1/` — 动态 API 查询(GraphQL、GraphQL 日志)
  * `rest/cbc/cbcbiportalservice/v1/` — 门户配置
  * `rest/cbc/cbcbiuserinfoservice/v1/` — 用户信息
  * `rest/cbc/cbcbipermissionservice/v1/` — 权限查询
    — 来源:`website-src/src/app/component/http/DataServiceHttpManager.ts:10-49`

### 1.2 基础路径

* **开发代理**:`/rest` → `https://100.95.135.220:18226`(内部地址);`/static` → `https://gray.cloudbi.huawei.com`
  — 来源:`website-src/proxy.conf.js:2-16`
* **上下文路径**:`/datav`
  — 来源:`website-src/src/environments/environment.ts:8`
* **生产环境**:无显式 baseUrl,所有 API 以相对路径 `/rest/...` 发起,由部署侧 Nginx 反向代理

### 1.3 鉴权方式

**关键发现:项目不使用 Bearer Token / Authorization Header。**

* **Token 获取**:

  * 生产环境:用户信息由宿主门户渲染注入到 `window["currentLoginUser"]` 全局变量,前端直接读取
    — 来源:`website-src/src/app/services/LoginService.ts:151`
  * 开发环境:硬编码 `loginUserId: 'm00420975'`,通过 `GET /rest/cbc/cbcbiuserinfoservice/v1/userinfo/{userno}` 查询
    — 来源:`website-src/src/app/services/LoginService.ts:24,50-54,58-59`
* **Token 传递**(通过 `HeaderSetterInterceptor` 自动注入):

| Header             | 值来源                                                            | 用途      |
| ------------------ | -------------------------------------------------------------- | ------- |
| `appId`            | `LoginService.getActiveAppId()` 或 `getEditAppId()`(按 URL 前缀切换) | 应用标识    |
| `tenantId`         | `LoginService.getCurrentTenant().tenantId`                     | 租户标识    |
| `x-operator-id`    | `LoginService.getCurrentLoginUser().userId`                    | 操作人     |
| `cftk`             | `localStorage.getItem("cftk")`                                 | CSRF 防护 |
| `x-language`       | 固定 `"zh-CN"`                                                   | 语言      |
| `X-Requested-With` | 固定 `"XMLHttpRequest"`                                          | AJAX 标识 |
| `Content-type`     | 固定 `"application/json;charset=utf-8"`                          | 内容类型    |

— 来源:`website-src/src/app/component/http/interceptor/header-setter.interceptor.ts:24-56`

* **身份鉴别实际依赖**:上游 SSO 的 Session Cookie(浏览器同域携带)+ 业务级 Header(`x-operator-id`/`tenantId`/`appId`)
* **登出处理**:HTTP 响应状态码为 `ANALYTICS_NOT_LOGIN` 时调用 `window.relogin()` 跳转重登
  — 来源:`website-src/src/app/component/http/interceptor/ErrorMessageInterceptor.ts:30-32`

### 1.4 超时与重试约定

**未发现任何超时配置或重试机制**。在 `website-src/src/app/component/http/` 全目录中未命中 `timeout`、`retry`、`TimeoutError`、`retryWhen` 等关键词。所有 HTTP 请求依赖 Angular `HttpClient` 默认行为(无超时、无重试)。

> **对新平台的启示**:新平台需自行实现超时(建议 30s)+ 重试(建议指数退避,最多 3 次,仅对幂等 GET)。

## 2. 自定义查询语言

### 2.1 总体架构

**表服务使用 GraphQL 作为底层查询语言,但前端并未直接使用标准 GraphQL,而是通过一套"指令(Directive)体系"在 GraphQL schema 之上封装了一层自定义扩展能力。**

* **API 入口**:所有 GraphQL 查询通过 POST 请求发送到 `/rest/cbc/cbcbidynamicapiservice/v1/graphql`,请求体格式为 `{"apiQuery": "{...}", "isTest": true}`
  — 来源:`website-src/@cloud/datav-lowcode/datasource/GraphQLDataSource.ts:26,300`;`website-src/src/app/component/http/GraphqlHttpManager.ts:17-23`
* **GraphQL 请求包装**:外层是一个 JSON 对象,`apiQuery` 字段值才是真正的 GraphQL 查询字符串
* **GraphQL introspection**:系统支持通过 `__type` introspection 查询表服务的元数据(字段列表)
  — 来源:`website-src/src/app/component/http/GraphqlHttpManager.ts:16-23`

### 2.2 表服务定义模式

表服务定义时支持两种模式:

* **SCRIPT 模式**:手写 SQL/XML 脚本,发布后自动映射为 GraphQL schema
* **CONFIG 模式**:通过 UI 配置表字段和参数
  — 来源:`website-src/src/app/model/cbcbidataservice/DataServiceModel.ts:98`

### 2.3 能力矩阵

#### 2.3.1 指令集(Directive)

系统定义了 6 种指令类型,通过 `GraphQLDirectiveType` 枚举:
— 来源:`website-src/@cloud/datav-lowcode/interface/datasource-interface.ts:8-15`

| 指令          | 作用域                  | 说明      | 示例                                         |
| ----------- | -------------------- | ------- | ------------------------------------------ |
| `@limit`    | 全局(Query)            | 限制返回行数  | `@limit(value:10)`                         |
| `@offset`   | 全局(Query)            | 分页偏移量   | `@offset(value:0)`                         |
| `@where`    | 全局(Query) / 表(Table) | 过滤条件    | `@where(value:"status = 'active'")`        |
| `@order`    | 字段(Field)            | 排序      | `@order(type:"asc", priority:1)`           |
| `@function` | 字段(Field)            | 聚合/衍生计算 | `@function(value:"sum", enhanced:true)`    |
| `@connect`  | 数据处理(DataProcess)    | 连接多表结果  | `@connect(value:"ds1Query\|ds1RestQuery")` |

#### 2.3.2 筛选操作符

`@where` 指令的 value 值为**类 SQL 条件表达式自由文本**(由后端解析),从代码示例可见支持:

* 比较运算符:`=` / `!=` / `>` / `<` / `>=` / `<=`
* `in` 语法 — 来源:`website-src/src/app/component/catalog/data-factory/api-service-editor/api-service-samples.ts:92` (`cloudservicetypecode in (#{cloudservicetypecode})`)
* `like` 语法 — 来源:`api-service-samples.ts:387` (`lower(creator_account) like CONCAT('%',LOWER(#{creator_account}),'%')`)
* `between` 语法 — 来源:`api-service-samples.ts:87`
* SQL 函数:`coalesce()`、`case when`、`lower()`、`CONCAT()`
  — 来源:`website-src/src/app/component/catalog/lowcode-example/data-source-config.ts:244-285`

> **注意**:`@where` 值为自由文本,具体支持的操作符取决于后端解析器,前端无校验。

#### 2.3.3 聚合方式

通过 `@function` 指令实现:

* 支持 `value` 参数传入聚合函数名(如 `sum`、`avg`、`count` 等,具体由后端 GraphQL schema 决定)
* 支持 `enhanced` 参数,开启增强计算模式
  — 来源:`website-src/@cloud/datav-lowcode/datasource/GraphQLDataSource.ts:465-473`
* **内置保留字段**:

  * `cnt` — 行计数
  * `{column}_sum` — 数值列自动求和(如 `metric_value_sum`)
    — 来源:`website-src/src/app/component/catalog/data-factory/data-table-editor/data-table-editor.component.ts:1096-1098`

#### 2.3.4 时间粒度

时间粒度在**数据服务定义阶段**通过 `DataParamOptions` 配置,而非查询时动态指定:

* 支持年/月/周/日/时/分/秒七种粒度
* 每种粒度有对应的格式模板:YYYY / YYYY-MM / YYYY-WW / YYYY-MM-DD / YYYY-MM-DD hh 等
  — 来源:`website-src/src/app/component/catalog/data-factory/data-table-editor/data-param-contant.ts:1-96`

> **对新平台的启示**:查询时动态指定时间粒度是常见需求,需要新平台在适配层做转换(把 DSL 中的 `granularity: "month"` 翻译为对预定义字段的引用)。

#### 2.3.5 排序

* 支持 `asc` / `desc` 两种排序方向
* 支持多字段排序:每个字段独立配置 `@order` 指令,通过 `priority` 参数控制优先级
* 排序指令可通过 `condition` 条件动态启用/禁用(支持全局变量模板)
  — 来源:`website-src/@cloud/datav-lowcode/datasource/GraphQLDataSource.ts:456-462`;`website-src/src/app/component/catalog/data-factory/debugger-graphql/debugger-graphql.component.ts:46-52`

#### 2.3.6 分页

* 使用 `@limit(value:N)` + `@offset(value:N)` 组合
* **不支持 cursor 游标模式**
* 分页参数支持模板表达式(如 `{{globalVar.pageSize}}`)
  — 来源:`website-src/@cloud/datav-lowcode/datasource/GraphQLDataSource.ts:488-493`

#### 2.3.7 TopN

* **无独立 TopN 指令**
* 可通过 `@order` + `@limit` 组合实现 TopN 效果

#### 2.3.8 同环比/衍生计算

**前端层面提供两种计算能力**:

1. **计算字段**(CalcColumn):使用 `Arithmetic(var1)` 四则运算和 `Grading(var1,var2)` 定级函数

   * 示例:`Arithmetic(90*({a.c} + {a.c})/{a.c})`
   * 示例:`Grading({a.sum},{b.evaluation_criterion})`
     — 来源:`website-src/src/app/component/catalog/data-factory/data-table-editor/data-table-editor.component.ts:306-307`
2. **混合计算字段**(RestCalcColumn):跨表关联的计算,需要 `joinExpression`
   — 来源:`data-table-editor.component.ts:1085-1091`

**同环比/同比计算**:**未在代码中发现内置的同环比指令或函数**,需通过 SQL 脚本层(`case when`/`coalesce`)或前端适配函数实现。
— 来源:`website-src/src/app/component/catalog/lowcode-example/data-source-config.ts:244-285`(用 SQL `case when` 手工计算同比)

### 3.2 统一数据返回格式

#### 3.2.1 通用响应信封

所有 REST API 遵循统一结构:

```typescript
interface BaseResponseInterface extends ExtraType {
  retCode?: string;        // 返回码,成功为 "CBC.0000"
  retDesc?: string;        // 返回描述
```

### 4.5 元数据接口缺失评估与补建成本

| 目录类型     | 现状                                          | 缺失程度     | 补建建议                                                                                                                                           |
| -------- | ------------------------------------------- | -------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| **指标目录** | 无独立接口。`MetricBaseInfo` 仅存在于 GraphQL 层的示例查询中 | **完全缺失** | 后端已有 `MetricBaseInfo` resolver(返回 `metric_code`、`metric_name_zh`、`scope`),前端需封装通用方法;若需"可用维度"、"可用聚合方式"等关联字段,后端需扩展 resolver 或在 ADS 设计层补充映射表      |
| **维度目录** | 无独立接口。维度仅作为卡片子字段存在                          | **完全缺失** | 需新建后端端点。可复用 `MetadataModel` 模型结构(`columnName`、`zhName`、`enName`、`dataType`、`enumable`、`enumDictId`);前端新建 `MetadataHttpManager`                 |
| **表目录**  | 表服务列表 + GraphQL `__type` 内省 + 标签中心三层下钻      | **部分具备** | `services/list` 接口可直接作为表目录使用(返回 `serviceCode`、`serviceName`、`description`、`paramOutputMetaData`)。GraphQL 内省可补充字段级元数据。无需新建接口,但需在前端统一封装为"目录查询"能力 |

**补建成本估算(待确认)**:

1. **指标目录**:后端已有基础 resolver,前端封装成本低(约 1-2 人日)。但 `可用维度` 和 `可用聚合方式` 需要后端扩展或在 ADS 设计层建立指标-维度-聚合的映射关系(3-5 人日)。
2. **维度目录**:需从零新建后端端点 + 前端封装(约 3-5 人日)。`BizMonitorCardModel` 中的 `AggrFunc`/`Algorithm`/`FilterOperator` 类可作为聚合方式和过滤算子的参考模型。
3. **表目录**:已有能力可直接复用,只需前端统一封装(1-2 人日)。

> **这是新平台的硬依赖**。MetricCanvas 的领域 DSL 中"指标+维度+筛选+聚合的 JSON 对象"必须有元数据目录支撑,否则用户无法知道有哪些指标/维度可用、单位是什么、支持哪些聚合方式。

### 5.3 每页的查询模式

从 `data-source-config.ts` 和两个 lowcode 页面配置中归纳出三种数据源类型:

| 数据源类型       | 说明                                             | 典型场景                   |
| ----------- | ---------------------------------------------- | ---------------------- |
| **graphql** | 通过 GraphQL 表服务查询,支持多表联合、字段过滤、where 条件、SQL 函数计算 | IOC 收入汇总、目标完成率、非员工人数查询 |
| **static**  | 静态 JSON 数据,开发调试用                               | 测试、Demo                |
| **api**     | REST API 数据源,支持参数模板绑定                          | 外部 API 集成              |

**GraphQL 查询模式特点**:

* 单表查询为主,支持 `where` 条件(如 `mtime='202211'`、`metric_code='revenue-external_customer'`)
* 支持 SQL 函数计算列:`coalesce`、`case when`,用于计算同比/环比/差值
* 支持 `orderDesc` 排序和 `limit` 限制
* 支持 `resultType: "arrayMap"` 返回数组映射格式
  — 来源:`website-src/src/app/component/catalog/lowcode-example/data-source-config.ts:163-378`

**页面查询模式分类**:

* **单指标卡展示**:`indicator` 组件绑定单一字段如 `metric_value_sum`

## 6. 权限与 SSO

### 6.1 登录接入方式

**结论:自研 SSO + 宿主框架注入,通过 single-spa/qiankun 微前端接入。**

* 生产环境中,用户信息不通过前端发起登录请求,而是由宿主框架注入到全局 `window["currentLoginUser"]` 对象中:

  ```typescript
  this.currentLoginUser = window["currentLoginUser"];
  ```

  — 来源:`website-src/src/app/services/LoginService.ts:151`
* 本地开发环境(非 production)下,前端通过 REST 接口查询用户信息:

  ```
  GET /rest/cbc/cbcbiuserinfoservice/v1/userinfo/{userno}
  ```

  — 来源:`website-src/src/app/services/LoginService.ts:24,58-59`
* 项目是一个 **single-spa + qiankun 双兼容的微前端子应用**:

  ```typescript
  const lifecycles = singleSpaAngular(userOpts);
  export const { bootstrap, mount, unmount } = lifecycles;
  if((window as any).__POWERED_BY_QIANKUN__) { ... }
  ```

  — 来源:`website-src/src/main.single-spa.ts:30-39`
* 同时存在独立的 `main.ts`(非微前端模式启动)
  — 来源:`website-src/src/main.ts:15`
* **无 OAuth/CAS/SAML 协议栈**。登录认证完全由宿主框架(门户)负责
* 用户对象模型中 `iamAuthType`、`iamAccountId` 字段暗示后端对接了 IAM
  — 来源:`website-src/src/app/model/cbcbipermissionservice/UserModel.ts:11-13`
* **未登录重定向**:拦截器检测到 HTTP 401 状态码(`ANALYTICS_NOT_LOGIN`)时调用宿主提供的 `window.relogin()`:

  ```typescript
  if (response.status === HttpRetCode.ANALYTICS_NOT_LOGIN) {
      (window as any).relogin();
  }
  ```

  — 来源:`website-src/src/app/component/http/interceptor/ErrorMessageInterceptor.ts:30-32`
* **CSRF 防护**:通过 `cftk` localStorage 值,在请求头 `cftk` 中传递
  — 来源:`website-src/src/app/component/http/interceptor/header-setter.interceptor.ts:52-54`

### 6.2 页面级权限配置

**结论:无 Angular Route Guard,权限控制通过后端权限服务 + 菜单动态渲染实现。**

* 路由表(`app-routing.module.ts`)**没有配置任何 `canActivate`/`canActivateChild` 守卫**,所有路由均对外开放
  — 来源:`website-src/src/app/app-routing.module.ts:10-45`
* 页面权限通过**权限中心 REST 服务**(`cbcbipermissionservice`)管理,登录时查询用户权限列表:


  — 来源:`website-src/src/app/services/LoginService.ts:23,145-148`
* 权限模型为 **RBAC: 角色(Role) → 应用(MicroApp) → 菜单(MenuAuthority) → 子菜单** 层级结构
  — 来源:`website-src/src/app/model/PermissionModel.ts:2-66`

### 6.3 数据级权限

**结论:数据权限由后端权限服务管控,前端仅做 UI 展示和管理;通过"控制字段(ControlField)"实现行级数据过滤。**

* **控制字段(ControlField)** 是实现数据级权限的核心机制。每个用户对其有权限的资产(DATA 或 APP 类型)可以绑定一组控制字段(字段名+字段值),后端在查询数据时根据这些字段值做行级过滤
  — 来源:`website-src/src/app/component/catalog/controlfield-manage/modal/detail/detail-modal.ts:44-50`
* 控制字段管理 API:

  ```
  POST   /rest/cbc/cbcbipermissionservice/v1/anaperm/getuserpermfields/{permissionId}  // 查询
  POST   /rest/cbc/cbcbipermissionservice/v1/anaperm/createuserpermfields             // 创建
  PUT    /rest/cbc/cbcbipermissionservice/v1/anaperm/updateuserpermfields/{id}        // 修改
  DELETE /rest/cbc/cbcbipermissionservice/v1/userpermissions/{userpermissionid}        // 删除
  GET    /rest/cbc/cbcbipermissionservice/v1/anaperm/getpermfields/{permissionId}      // 按权限查字段列表
  ```

  — 来源:`website-src/src/app/component/catalog/controlfield-manage/components/PermissionHttpManager.ts:103-107,853-875`
* 权限类型分为 `DATA`(数据卡)和 `APP`(应用)两种
  — 来源:`website-src/src/app/component/catalog/controlfield-manage/constant/permission-type.ts:1-11`
* 用户资产权限表中包含 `applybeginTime`/`applyendTime`(生效/失效时间)、`accessmode`(权限获得方式)等字段,支持基于时间窗口的权限控制
  — 来源:`website-src/src/app/component/catalog/controlfield-manage/controlfield-manage.component.ts:48-50`
* **前端不做数据过滤**,控制字段值存储在权限服务中,由后端数据查询时生效

### 6.4 租户/应用隔离

* 每个 HTTP 请求自动注入 `tenantId`、`appId`、`x-operator-id` 请求头
  — 来源:`website-src/src/app/component/http/interceptor/header-setter.interceptor.ts:27-38`
* 租户模型包含 `tenantId`、`tenantName`、`customerId`、`customerName`、`authType`(租户 Token/系统 Token)
  — 来源:`website-src/src/app/model/cbcbipermissionservice/RoleModel.ts:17-44`
* **用户-角色-租户关联**:RoleModel 中 `tenantIds` 数组表示该角色绑定的租户列表;用户登录后,系统从用户所有角色的 `tenantIds` 中收集出可用租户列表
  — 来源:`website-src/src/app/services/LoginService.ts:232-245`
* 部分 API 路径中直接嵌入 `tenantId` 作为路径参数,如 `rest/cbc/cbcbidataservice/v1/dataservice/{tenantId}/card`
  — 来源:`website-src/src/app/component/http/CardHttpManager.ts:29-48`

  — 来源:`website-src/src/app/component/http/interceptor/header-setter.interceptor.ts:27-28`

### 6.5 对 MetricCanvas 的启示

1. **认证协议**:当前项目无独立登录页,完全依赖宿主门户注入 `window.currentLoginUser`。MetricCanvas 需确认新平台的 SSO 接入方式 —— 如沿用 IAM 体系则需对接相同的用户注入机制;若独立部署,则需实现 OAuth2/OIDC 或 CAS 流程。
2. **页面权限**:当前无前端路由守卫,依赖后端权限服务返回菜单树做动态渲染。MetricCanvas 可考虑引入 Route Guard 作为前端防御层,同时后端保持 `cbcbipermissionservice` 兼容。
3. **数据权限**:控制字段(ControlField)机制是行级数据过滤的核心。MetricCanvas 的数据查询需识别 `tenantId` + `x-operator-id` 请求头,将控制字段过滤下推到 SQL 生成层或表服务层,**而不是前端过滤**。
4. **租户隔离**:`tenantId` + `appId` 双维度隔离,租户信息来自角色关联的 `tenantIds`。MetricCanvas 需保持请求头兼容,并在数据层按租户做物理或逻辑隔离。

## 8. 意外发现

### 8.1 GraphQL 查询体被 JSON 双重包装

GraphQL 查询字符串不是直接作为 POST body,而是被包装在 `{"apiQuery": "...", "isTest": true}` 中。这意味着:

* 标准 GraphQL 客户端(Apollo Client 等)**无法直接对接**,需要自定义 fetcher
* `isTest: true` 字段含义待确认 —— 推测用于区分测试模式与生产模式(可能影响缓存、日志等)
  — 来源:`website-src/src/app/component/http/GraphqlHttpManager.ts:17-23`

### 8.2 批量查询上限为 5

`GraphqlGroupEngine` 中硬编码 `MAX_BATCH_SIZE = 5`,超过会拆分。如果一个看板页面同时发起 10+ 个查询,会产生多次 HTTP 往返。
— 来源:`website-src/@cloud/datav-lowcode/datasource/GraphqlGroupEngine.ts:10`

### 8.3 响应不返回总条数,前端无法实现"共 N 条"

分页响应中**没有 total 字段**,前端只能盲翻或单独发 count 请求。`TiPageConfig` 中的 `totalNumber` 字段需要前端自行维护。
— 来源:`website-src/src/app/common/interfaces/TiPageConfig.ts:1-17`

### 8.4 时间粒度在服务定义时固定,查询时无法动态切换

时间粒度(年/月/周/日)在数据服务定义阶段通过 `DataParamOptions` 配置,**查询时无法动态指定**。MetricCanvas 如果要支持"用户切换时间粒度",需要在适配层做转换:把 DSL 中的 `granularity: "month"` 翻译为对预定义字段的引用,或者要求表服务后端支持查询时指定粒度。
— 来源:`website-src/src/app/component/catalog/data-factory/data-table-editor/data-param-contant.ts:1-96`

### 8.5 JOIN 在服务定义时配置,查询时无法动态 JOIN

虽然支持 5 种 JOIN 类型,但关联关系在**数据服务创建时**就固定了。MetricCanvas 如果要支持"用户在查询时动态选择维度下钻",可能需要预先定义好所有可能的 JOIN 关系,或者要求表服务后端支持查询时动态 JOIN。
— 来源:`website-src/src/app/component/catalog/data-factory/data-table-editor/related-table/related-table.component.ts:20-26`

### 8.6 同环比无内置支持,依赖 SQL `case when` 手工计算

代码中看到的同环比计算都是通过 SQL `case when` + `coalesce` 在数据服务定义层手工实现的。这意味着:

* 每个同环比指标都需要在数据服务定义时写一段 SQL
* 用户无法在查询时动态指定"对比去年同期"或"对比上月"
* MetricCanvas 如果要支持灵活的同环比,需要在适配层封装一套同环比计算模板,生成对应的 SQL 片段
  — 来源:`website-src/src/app/component/catalog/lowcode-example/data-source-config.ts:244-285`

### 8.7 `MetricBaseInfo` resolver 暗示后端有指标元数据能力,但前端未封装

`DataSourceHttpManager.queryRestList()` 方法中引用了 `MetricBaseInfo` resolver,可返回 `metric_code`、`metric_name_zh`、`scope`。这是一个**重要发现**:后端可能已经有指标元数据能力,只是前端没有封装成通用方法。MetricCanvas 团队应优先与后端确认这个 resolver 的完整能力,可能能大幅降低指标目录补建成本。
— 来源:`website-src/src/app/component/editor/http/DataSourceHttpManager.ts:41-48`

### 8.8 数据权限通过"控制字段"下推,前端不感知过滤逻辑

控制字段(ControlField)机制是行级数据过滤的核心,但**前端完全不感知过滤逻辑** —— 控制字段值存储在权限服务中,由后端数据查询时生效。这意味着:

* MetricCanvas 的查询请求中**不需要**显式传递数据权限参数
* 但需要确保请求头中携带 `x-operator-id` 和 `tenantId`,后端才能识别用户身份并应用控制字段过滤
* 如果新平台要支持"用户预览自己能看到的数据范围",需要单独调用权限服务查询控制字段配置
  — 来源:`website-src/src/app/component/catalog/controlfield-manage/modal/detail/detail-modal.ts:44-50`

### 8.9 微前端双兼容(single-spa + qiankun)

项目同时兼容 single-spa 和 qiankun 两种微前端框架:

```typescript
const lifecycles = singleSpaAngular(userOpts);
export const { bootstrap, mount, unmount } = lifecycles;
if((window as any).__POWERED_BY_QIANKUN__) { ... }
```

— 来源:`website-src/src/main.single-spa.ts:30-39`

如果 MetricCanvas 也要作为微前端子应用接入同一门户,需要保持同样的双兼容模式。`window.currentLoginUser`、`window.relogin()`、`window.messageInterceptor` 等全局变量是宿主与子应用之间的契约。

### 8.10 VSCode WebView 特殊通道

当 `environment.vscode === true` 时,HTTP 请求走 `VscodeHttpHandlerManager` 而非标准 HTTP,通过 ReplaySubject 桥接 Webview 消息。这暗示项目有 VSCode 插件形态(代码中确实存在 `vscode-plugin/` 目录)。MetricCanvas 如果不考虑 VSCode 集成,可以忽略此通道。
— 来源:`website-src/src/app/component/http/interceptor/header-setter.interceptor.ts:43-44`;`website-src/vscode-plugin/`

### 8.11 `@cloud/datav-lowcode` 是核心但不在本仓库

`@cloud/datav-lowcode` 是低代码引擎的核心包,但**不在本仓库源码中**(npm 依赖)。这意味着:

* MetricCanvas 如果要复用低代码引擎,需要协调包 owner
* 包内部实现无法直接审计,只能通过导出接口和外部使用方式推断能力
* 该包深度耦合 Angular,MetricCanvas 如果用其他框架(如 React/Vue)需要重写
  — 来源:`website-src/@cloud/datav-lowcode/`(本地缓存的 npm 包)

### 8.12 路径前缀中的版本号暗示 API 演进

REST 路径中可见 `v1` 和 `v2` 共存:

* `rest/cbc/cbcbidataservice/v1/service/test/api/{serviceCode}` (v1)
* `rest/cbc/cbcbidataservice/v2/service/test/api/{serviceCode}` (v2)

— 来源:`website-src/src/app/component/http/DataServiceHttpManager.ts:38-39`

说明后端在演进,MetricCanvas 设计时应预留 API 版本切换能力。
