# 领域层建模为规格语言(元领域),包按 DDD 分层围绕聚合根"看板页面"命名

平台自身的领域层不含传统业务实体,这是刻意的架构事实而非建模缺陷:指标、维度、口径、衍生计算的定义权全部在数据服务(ADR-0003 连同环比都下沉),平台拥有的领域概念只有"描述与编排"层面的词——页面规格、结构化查询、筛选状态、数据快照。平台是一个元层:描述页面的语言,如同 Terraform 的领域是 HCL 而非服务器。因此:

1. **领域包按聚合根命名为 `page`(看板页面)**,而非按介质命名(spec/schema)或硬造业务名(metrics/board)。页面是聚合根,页面规格是它的唯一真源,校验规则是它的不变式——包内容(类型+校验)与包名(聚合)一致且不撒谎。
2. **端口按意图命名,适配器按系统命名。** 应用层(`runtime`)持有 `DataGateway`(取数端口,生效查询进、数据行出)与 `SpecProvider`(规格加载端口);基础设施包 `data-gateway` 提供适配器(数据服务、mock)。依赖倒置同时体现在 import 方向与命名上——运行时的词汇里不出现任何具体数据系统。
3. **表现层包名 `widgets` 取自规格字段**,让读规格与读代码用同一个词。
4. **分层依赖规则:** 全部箭头指向领域层 `page`;`runtime` 定义端口,`data-gateway` 实现端口,`apps/canvas` 作为组合根注入。

同日决策:词汇表术语"表服务"正式更名"**数据服务 (Data Service)**"(历史调研报告《中间层分析.md》《组件分析.md》《origin.md》保留原文)。"数据服务是唯一数据入口"的约束不因 `data-gateway` 包名通用化而放松:新增数据源须经词汇表与 ADR 决策,不得绕过网关。

## Considered Options

- 领域包叫 `spec`/`spec-schema`(按介质):诚实但无指称,包名与聚合根类型 `PageSpec` 脱节,被否决。
- 领域包叫 `dashboard`/`大屏`:"大屏"会错误锚定到存量 datav 物种;`dashboard` 与 `page` 二选一时,词汇表术语是"页面规格",取 `page`。
- 适配器包保留 `table-service-client`(按系统):与"表服务→数据服务"更名冲突,且端口/适配器同名会破坏倒置语感,被否决。

## Consequences

- 领域层永远不会出现业务计算;若某天需要平台侧衍生指标,应先修订 ADR-0003 而不是往 `page` 包塞逻辑。
- 包名与词汇表绑定:词汇表术语变更(如本次"数据服务")需评估包/端口命名级联。
- `page` 将来独立发包(validate CLI)须用打包器产出 ESM(仓库内 import 不带扩展名,裸 tsc 编译产物不可用)。
