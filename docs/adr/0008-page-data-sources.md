# 公开页面采用命名数据源,统一 inline 与 query 取数

看板页面公开声明 `dataSources`。每个页面数据源同时给出 `fields` 输出契约和带判别字段的 `source`:查询数据使用 `source.type: "query"` 与结构化查询,固定时点数据使用 `source.type: "inline"` 与 `rows`。组件不携带查询或数据行,只通过 `data` 命名数据槽引用页面数据源；单源组件通常使用 `main`,多源组件可使用 `compare`、`target` 等语义槽位。

统一运行时直接消费看板页面,不再使用区分 `live` / `snapshot` 的渲染输入信封。inline 与 query 进入同一数据源编排路径:前者同步产生终态数据快照,后者把结构化查询与筛选状态合成为生效查询,经数据网关取数并维护加载、错误与竞态状态。同一字段契约可在 inline/query 之间切换,混合页面按组件实际绑定的数据源精确派生实时能力。

仅含 inline 页面数据源的页面是静态页面。校验器拒绝其 `filters`、actions 和远程分页,避免“界面已变化、数据未变化”的假交互；query 页面数据源可声明筛选订阅,组件 actions 仅在绑定 query 数据源时生效。元数据快照只校验 query source 的指标与维度供给,inline rows 则按自身字段契约校验字段完备性、值类型和多余字段。

## Consequences

- 固定报告、实时看板及两者混合的页面共享一个公开协议、一个页面仓储接口和同一套纯渲染组件。
- 页面数据来源与数据行在单份 JSON 中自描述；AI、校验器和评审者无需理解外部交付信封或按组件匹配快照键。
- inline rows 会随页面资产固化,不会自动刷新,也不具备筛选联动语义；需要实时数据时使用 query source。
- 页面可内嵌受字段契约约束的 inline rows,但仍禁止表达式、HTML、脚本、自定义样式和绕过数据网关的远程取数,ADR-0003 的严格声明式约束不变。

## Considered Options

- 保留 `live` / `snapshot` 渲染输入信封:使公开交付出现第二套顶层协议,并要求快照按组件键组织,被否决。
- 直接给每个组件增加数据行或结构化查询:把取数、字段契约和展示耦合到组件,无法复用数据集或表达多源组件,被否决。
- 把固定数据伪装成数据网关实现:会让静态来源藏在依赖注入中,单份页面无法自描述,被否决。
- 页面与固定数据拆成两个文件:模型清晰但不满足离线交付的单文件要求,保留为外部存储可采用的物理实现,不作为公开页面协议。
