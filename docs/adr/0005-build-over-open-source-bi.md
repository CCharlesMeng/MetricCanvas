# 自研看板运行时,不采用/魔改 Grafana、Superset、Rill 等开源 BI

一期交付物表面上与成熟开源 BI 高度重叠("声明式规格 + 看板运行时"),必须回答 build vs buy。我们决定自研,理由:

1. **数据源模型不匹配。** 开源 BI 的数据源假设是 SQL/时序数据库;我们的唯一数据入口是数据服务——一个指标/维度语义接口。接入要写厚重的数据源插件,且指标语义仍会被摊平成 SQL 结果集,丢掉"结构化查询可逐字段校验"这个 AI 可控性的根基。
2. **规格可控性是核心诉求,不是附带需求。** 开源 dashboard JSON 为人类编辑器设计,庞大、宽松、允许自由 panel options(Grafana schemaVersion 已演进至 42)。AI 生成这种格式不可控、难校验;我们需要的是封闭、紧凑、带语义校验的领域 DSL(ADR-0003)。
3. **魔改的长期成本高于自建。** fork 开源 BI 意味着永久跟随上游演进、在别人的架构里塞入数据服务语义和自己的联动模型;而自建运行时只需渲染一个封闭 BI 组件集,复杂度可控(Rill 单团队维护同架构是先例)。
4. **部署与整合约束。** 内网部署、公司 SSO、与数据服务的权限体系整合、iframe/Web Component 嵌入存量系统——深度定制在自有代码库中成本最低。

## Considered Options

- 直接采用 Grafana/Superset:数据源与规格模型不匹配,AI 可控性诉求无法满足,被否决。
- fork 后深度魔改:短期快、长期双倍维护,被否决。
- 采用其规格格式但自建运行时:格式为人类编辑器设计,不满足 AI 生成可控性,被否决。

## Consequences

- 承担运行时全部自研成本;组件集、DX 工具链没有免费午餐。
- 定期(每半年)回望开源 BI 与 AI-BI 生态演进,本 ADR 的前提若失效(如出现指标语义原生、AI 可控的开源规格标准)应重新评估。
