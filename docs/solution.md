# MetricCanvas(指标画布)整体解决方案

> 以表服务提供数据,以页面规格描述应用,以统一运行时渲染页面,以 AI 完成页面组合和数据分析。

本方案是对 `origin.md` 中架构调整目标的完整落地设计。术语以根目录 `CONTEXT.md` 为准,关键决策记录见 `docs/adr/`。

---

## 1. 项目定位

**表服务之上的治理化消费层:让指标以一致、可审计、AI 可操作的方式变成页面。**

价值锚点是**治理与 AI 可控性**,不是单纯的开发提效——2026 年 AI 直接生成高代码页面已经很快,规格化路线的不可替代性在于:指标口径一致、页面可 diff 可审计、变更可评审、AI 输出可校验可自动纠错、长期资产不随框架腐化。

- **服务对象(一期):** 开发者 / 数据工程师,AI 辅助生成与修改页面规格;后期逐步开放业务用户。
- **与存量关系:** 新页面用新平台;存量 Angular 页面维持不动、自然淘汰,不做迁移立项。
- **数据边界:** 表服务(GraphQL 之上)以下的数据底座完全不动,表服务是平台唯一数据入口。

核心资产从"页面源代码"转变为"页面规格":AI 生成和修改的是结构化、可校验的规格文档,而不是不可控的生产代码。自研运行时而非采用 Grafana/Superset 等开源 BI 的理由见 ADR-0005。

## 2. 架构分层

```text
┌─────────────────────────────────────────────────┐
│  消费层                                          │
│  独立站点(SvelteKit SPA) / iframe / Web Component │
├─────────────────────────────────────────────────┤
│  统一运行时(Svelte + shadcn-svelte + ECharts)     │
│  规格解析 → 布局渲染 → 组件实例化 → 筛选状态管理     │
├─────────────────────────────────────────────────┤
│  页面规格(领域 DSL,核心资产)                      │
│  一期: Git 仓库文件   长期: 平台后端 + 数据库        │
├─────────────────────────────────────────────────┤
│  规格生产                                        │
│  一期: 开发者手写 + AI 辅助                        │
│  二期: MCP 工具(查元数据/校验/保存) + 内置对话       │
├─────────────────────────────────────────────────┤
│  表服务(不动)                                    │
│  自定义查询语言 / 指标·维度·聚合查询 / 标准化 API    │
├─────────────────────────────────────────────────┤
│  GraphQL / ADS / DWS·DWD / ODS / 数据湖(不动)     │
└─────────────────────────────────────────────────┘
```

各层职责单一:规格只描述"是什么",运行时决定"怎么渲染",表服务决定"数据怎么算"。

主架构图只画现实存在的东西;A2UI 等未立项的演进方向见文末[附录:演进方向](#附录演进方向)。

## 3. 页面规格(领域 DSL)

设计原则(ADR-0001、ADR-0003):

1. **严格声明式。** 只表达查询引用、布局、图表类型、筛选联动;禁止表达式、脚本、自定义样式。同环比、衍生计算等一律下沉到表服务的指标定义。
2. **以 BI 语义为一等公民。** 指标、维度、聚合、筛选是规格的原生概念,不是通用组件树上的挂件。
3. **可完整校验。** JSON Schema 结构校验 + 对表服务元数据的语义校验(指标/维度是否存在、聚合是否合法),两级校验为 AI 生成提供自动纠错回路。
4. **带版本。** `specVersion` 字段。规格存 Git 意味着历史文件长期存在,不能假设"一次性迁移全量":运行时同时支持当前与前一个大版本(N / N-1),CI 对 N-1 规格给出升版警告,迁移脚本按需执行;N-2 才拒绝加载。

### 规格结构示例

```json
{
  "specVersion": "1.0",
  "id": "sales-overview",
  "title": "销售总览",
  "filters": [
    { "id": "f-time", "type": "timeRange", "default": "last30d" },
    { "id": "f-region", "type": "dimension", "dimension": "region" }
  ],
  "layout": { "type": "grid", "columns": 12 },
  "widgets": [
    {
      "id": "w-gmv",
      "type": "metricCard",
      "position": { "x": 0, "y": 0, "w": 3, "h": 2 },
      "query": {
        "metrics": ["gmv"],
        "filters": { "subscribe": ["f-time", "f-region"] }
      }
    },
    {
      "id": "w-trend",
      "type": "lineChart",
      "position": { "x": 3, "y": 0, "w": 9, "h": 4 },
      "query": {
        "metrics": ["gmv", "order_count"],
        "dimensions": ["date"],
        "granularity": "day",
        "filters": { "subscribe": ["f-time", "f-region"] }
      }
    },
    {
      "id": "w-by-region",
      "type": "barChart",
      "position": { "x": 0, "y": 4, "w": 6, "h": 4 },
      "query": {
        "metrics": ["gmv"],
        "dimensions": ["region"],
        "filters": { "subscribe": ["f-time"] }
      },
      "interactions": [
        { "on": "click", "writeFilter": "f-region", "value": "$dimension.region" }
      ]
    }
  ]
}
```

### 数据引用:结构化查询

规格中不出现查询语句字符串。每个图表声明一个结构化查询对象(指标 + 维度 + 筛选 + 聚合粒度),运行时翻译为表服务请求。好处:逐字段校验、AI 生成可控、表服务查询语言演进不影响规格资产。

### 联动模型:页面级筛选状态 + 跨页下钻

页内联动:

- 页面声明若干筛选器(维度筛选、时间筛选),构成共享筛选状态;
- 图表通过 `filters.subscribe` 声明订阅哪些筛选器;
- 图表交互(如柱状图点击)通过 `interactions.writeFilter` 回写筛选状态,实现联动下钻;
- 组件之间不直接连线,所有联动只通过筛选状态传递,复杂度封顶。

跨页下钻(高频刚需,规格一等公民):从总览页图表跳转到明细页并携带筛选条件,用 `navigate` 交互声明:

```json
{
  "on": "click",
  "navigate": {
    "page": "sales-detail",
    "carryFilters": ["f-time"],
    "setFilters": { "f-region": "$dimension.region" }
  }
}
```

`carryFilters` 把当前页筛选状态带到目标页同名筛选器,`setFilters` 用点击上下文写入目标页筛选器。跨页传递的物理载体是 URL(筛选状态可序列化、可分享),目标页按第④步生命周期从 URL 恢复。校验器检查目标页存在、且目标筛选器 id 有效。

## 4. 统一运行时

技术栈(ADR-0002):

| 项 | 选型 | 说明 |
|---|---|---|
| 框架 | Svelte 5 + SvelteKit(static/SPA 模式) | 不用 SSR,构建产物为静态站点 |
| UI 组件 | shadcn-svelte + Tailwind CSS | 社区移植版,组件代码归我们所有 |
| 图表 | ECharts | 框架无关、图表类型全、含地图 |
| 布局 | 12 列网格 | 组件声明 x/y/w/h,与 Grafana/Rill 同款模型 |
| 校验 | JSON Schema(Ajv 或同类) | 结构校验;语义校验调表服务元数据接口 |

### 运行时职责

1. **规格加载:** 统一抽象接口 `SpecProvider`(一期实现:静态文件;二期实现:平台 API)。从第一天起运行时只依赖该接口,存储切换不改运行时。
2. **规格校验:** 加载后先过两级校验,失败给出结构化错误(供人看,也供二期 AI 修复回路用)。
3. **筛选状态管理:** 页面级 store,是联动的唯一总线:筛选器写入、图表订阅、交互回写,组件间不直接对话。
4. **查询编排:** 有状态调度台。将 widget 的结构化查询与其订阅的筛选器当前值合成**生效查询**,做去重、缓存、并发调度后交给表服务适配层;结果包装为**数据快照**(数据 + 就绪/加载/错误/空态)分发给组件。
5. **组件渲染:** 封闭组件集,按规格实例化。

### 页面生命周期(概念模型)

一个页面从打开到联动的完整流程,每个模块职责唯一:

```text
① 应用壳:路由解析 pageId,SSO/权限检查
② SpecProvider:load(pageId) → 页面规格
③ spec-schema:validate(规格, 元数据目录),不通过则渲染错误页
④ 筛选状态:按规格 filters 声明初始化(URL 有值则恢复)
⑤ 查询编排器:每个 widget 的结构化查询 × 订阅的筛选器当前值
   ═合成═> 生效查询;去重/缓存/并发调度
⑥ 表服务适配层:生效查询 → 表服务请求 → 归一化数据(无状态纯翻译)
⑦ 查询编排器:包装为数据快照,分发给各 widget
⑧ 纯渲染组件:按 props 渲染;用户交互 → 事件上抛
⑨ 运行时:按规格 interactions 将事件写回筛选状态 → 回到⑤
   (仅订阅了该筛选器的 widget 重新计算)
```

**编排器与适配层的分界:** 编排器决定"何时查、查什么、能不能合并",适配层只负责"一条生效查询怎么变成一次表服务请求"。适配层无状态、不感知页面概念,因此可被运行时之外复用:CI 语义校验的元数据目录、二期 MCP 工具、三期 AI 分析取数,用的都是它。

### 组件纯渲染原则

组件只渲染"静态数据",一切"动态"归运行时:

- 组件通过 props 接收两样东西:**数据快照**(运行时查询编排的结果)和**展示配置**(来自规格);
- 组件内部**不发请求、不访问全局状态、不管理加载/错误/空态**——这些统一由运行时的查询编排层处理,组件拿到的永远是就绪的数据;
- 组件对外的唯一输出是**交互事件上抛**(如柱状图点击),由运行时决定事件如何回写筛选状态,组件不感知联动逻辑;
- 筛选器组件同样纯渲染:显示运行时传入的候选项与当前值,变更时只上抛事件,不直接写筛选状态。

收益:组件可脱离运行时独立开发和测试(mock 一份数据快照即可)、图表库可替换、组件语义清晰,未来接 A2UI 时能直接映射为 catalog 组件定义。

### 一期组件集

已按存量看板使用统计校准(调研报告《中间层分析.md》§5.2):

- **展示:** 指标卡、折线图、柱状图、饼图、表格、环形进度、地图、文本(标题/说明/带参链接)
- **筛选:** 维度筛选器、时间范围筛选器

表格是存量使用频率最高、交互最重的组件(下钻、内嵌趋势),为一期复杂度重点;表服务响应不含总条数,分页 UI 按盲翻或单独 count 设计。

组件集封闭是刻意的:运行时不是通用 UI 引擎,新组件类型走平台版本发布,不走页面级扩展。

## 5. 治理与工程流程

### 一期:规格即代码(ADR-0004)

- 规格文件存 Git 仓库,目录即命名空间;
- PR 评审 = 页面发布评审;CI 跑两级校验,校验不过不能合并;
- 合并后静态部署,运行时按路由加载对应规格;
- 权限:登录态沿用现有体系(宿主门户注入用户 + Session Cookie,请求头携带 `x-operator-id`/`tenantId`/`appId`/`cftk`,详见《中间层分析.md》§6);页面级访问控制一期用路由级配置实现;数据行级权限由后端"控制字段"机制下推,前端零感知。

### 长期:平台化

规格入库,平台后端提供版本历史、发布状态、权限 API;建设可视化管理界面(规格列表、预览、在线编辑、发布管理)。运行时无需改动(只换 `SpecProvider` 实现)。

## 6. 三期路线图

三期共享一条主线(表服务 → 规格 → 运行时),但**每期有独立价值和止损点**:任何一期结束后项目中止,已交付的东西仍然自洽可用,不依赖后一期兑现价值。

### 一期:地基(DSL + 运行时)

**目标:开发者手写规格,能渲染出生产可用的看板页面。**
**独立价值:治理化看板——口径一致、可 diff、可评审、可审计的看板资产与发布流程。**
**止损点:即使二期不启动,一期成果作为"配置化看板工具"仍可长期服役。**

- DSL schema v1(含 JSON Schema 定义)
- 统一运行时(加载、校验、筛选状态、查询编排、10 个组件)
- 表服务查询适配层 + 元数据接口对接
- Git 工作流 + CI 校验 + 静态部署 + SSO 接入
- 自动索引页:CI 扫描规格仓库生成看板目录首页(标题/描述/分组),平台的"发现"入口
- 用 2~3 个真实业务看板验收

**验收标准(对照组是"用 AI 直接编写高代码页面",不是手写 Angular):**

1. 交付速度不输 AI 直写代码;
2. 治理增益显著:同一指标在所有页面口径一致、每次变更可 diff 可评审、规格可复用为模板;
3. 验收看板覆盖真实需求,而非挑软柿子(缺口按预定程序处理,见 ADR-0003 的边界约束)。

### 二期:AI 生成规格

**目标:AI 通过对话生成和修改页面规格。**
**独立价值:AI 搭建能力本身即可对外交付,不依赖三期。**
**止损点:MCP 工具即使内置对话不做,也已让全公司任意 Agent 具备搭看板能力。**

- 平台暴露 MCP 工具:`查询指标/维度元数据`、`校验规格`、`保存规格`、`预览页面`
- 任意 Agent(Cursor、内部 Agent 平台等)可接入;平台内置对话界面只是其中一个客户端
- 生成回路:检索表服务元数据 → 生成规格 → 校验 → 错误回传自动修复 → 预览确认
- 平台后端 + 规格入库 + 可视化管理界面在本期落地

### 三期:AI 数据分析

**目标:从"AI 搭页面"到"AI 做分析"。**
**独立性声明:本期核心(对话式取数)只依赖表服务 + MCP 元数据/查询工具,不依赖一二期的 DSL 与运行时;若公司优先级要求,本期可提前独立启动。**

- 对话式取数与归因分析(基于表服务的结构化查询,不生成 SQL)
- 分析结论沉淀为规格片段,一键插入看板(此项依赖一二期)
- 探索生成式 UI:按对话即时生成临时分析视图
- 规格缺口反哺指标定义:`METRIC_GAP` 信号(一期为 issue 模板,二期为 MCP 自动开 issue)进入平台闭环,平台成为暴露指标需求的雷达,而不只是消费存量指标

## 7. 一期搭建清单

按依赖顺序:

0. **第 0 号任务——元数据地基:** 调研已完成(《中间层分析.md》§4):指标/维度目录接口**缺失**,表目录部分具备,后端存在未封装的 `MetricBaseInfo` resolver。走"残缺→共建"路线:快照初版由 `services/list` + GraphQL `__type` 内省 + `MetricBaseInfo` 拼装;并与后端确认扩展"可用维度/可用聚合"字段。落地 `sync-catalog` + 元数据快照机制。**全链单点,前置于一切。**
1. **DSL 定义包 `spec-schema`:** TypeScript 类型 + JSON Schema + 校验器(结构校验、语义校验接口),独立 npm 包,附 `validate` CLI(本地/pre-commit/CI 同一套逻辑)。校验错误分类型:`SCHEMA_ERROR`(规格写错,改规格)与 `METRIC_GAP`(业务想要、表服务没有——是需求信号,不是错误)。
2. **表服务适配层 `table-service-client`:** 结构化查询 → 表服务请求的翻译器;元数据接口(指标/维度目录)封装;`sync-catalog` 命令生成入库的元数据快照(CI 语义校验的确定性参照,业务数据永不落盘);附 mock 实现(按快照造形状正确的假数据,供离线开发与演示)。
3. **运行时应用 `runtime`:** SvelteKit 工程,`SpecProvider` 抽象 + 静态文件实现,筛选状态 store,查询编排器,10 个组件(shadcn-svelte + ECharts);dev server watch `specs/` 目录,改规格秒级热刷新。
4. **规格仓库结构:** `specs/` 目录 + CI 校验流水线 + 静态部署配置 + 自动索引页生成;CI 遇 `METRIC_GAP` 时引导按模板开 issue(label:`metric-gap`),定期同步数仓/表服务团队。
5. **SSO 与页面级权限。**
6. **验收看板:** 挑 2~3 个存量 Angular 看板中有代表性的,用规格重写对比。

## 附录:演进方向

**指标供需闭环(占位,细化待定)。** 场上存在三份指标清单、三个时态:**指标需求清单**(需求侧,将来时——需求讨论产出"页面要什么指标、含义、呈现在哪",由独立数据流程消费,寻找数据源并落成表服务指标)、**元数据快照**(供给侧,现在时——表服务有什么)、**页面规格**(消费侧,进行时——页面在用什么)。`METRIC_GAP` 即需求与供给的差集。两个待细化的配合点:① 三方使用同一套指标命名规范,"需求清单中哪些已落库、哪些是缺口"即可自动 diff;② 规格 widget 反向标注对应的需求条目,形成"业务需求 → 指标 → 页面呈现"的完整审计链。待表服务提炼报告返回后展开设计。

**A2UI 第二渲染通道(未立项)。** 页面规格是唯一资产,渲染通道可以多路并存。当 A2UI 协议成熟(v1.0 定稿、生态稳定)后,可立项"A2UI 适配器"——把同一份规格翻译成 A2UI 消息流,供支持 A2UI 的 Agent 宿主直接渲染,无需部署我们的运行时。规格不感知渲染通道,这是 ADR-0001 不把资产绑定协议的架构收益:接入 A2UI 是"加一个翻译器",不是"改资产格式"。组件纯渲染原则保证组件集可映射为 A2UI 自定义 catalog。评估节点放在三期。
