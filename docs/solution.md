# MetricCanvas(指标画布)整体解决方案

> 以表服务提供数据,以页面规格描述应用,以统一运行时渲染页面,以 AI 完成页面组合和数据分析。

本方案是对 `origin.md` 中架构调整目标的完整落地设计。术语以根目录 `CONTEXT.md` 为准,关键决策记录见 `docs/adr/`。

---

## 1. 项目定位

一套 **AI 原生数据应用平台**,替代现有 Angular 高代码看板开发模式。

- **服务对象(一期):** 开发者 / 数据工程师,AI 辅助生成与修改页面规格;后期逐步开放业务用户。
- **与存量关系:** 新页面用新平台;存量 Angular 页面维持不动、自然淘汰,不做迁移立项。
- **数据边界:** 表服务(GraphQL 之上)以下的数据底座完全不动,表服务是平台唯一数据入口。

核心资产从"页面源代码"转变为"页面规格":AI 生成和修改的是结构化、可校验的规格文档,而不是不可控的生产代码。

## 2. 架构分层

```text
┌─────────────────────────────────────────────────┐
│  消费层                                          │
│  独立站点(SvelteKit SPA) / iframe / Web Component │
│  [未来] 支持 A2UI 的 Agent 宿主                    │
├────────────────────────┬────────────────────────┤
│  渲染通道①(当前唯一)     │  渲染通道②(未来可插拔)   │
│  统一运行时              │  A2UI 适配器            │
│  Svelte + shadcn-svelte │  规格 → A2UI 消息流,     │
│  + ECharts              │  供 Agent 宿主端渲染     │
│  规格解析→布局→组件→联动  │  (协议成熟后再立项)      │
├────────────────────────┴────────────────────────┤
│  页面规格(领域 DSL,核心资产)                      │
│  一期: Git 仓库文件   长期: 平台后端 + 数据库        │
├─────────────────────────────────────────────────┤
│  规格生产                                        │
│  一期: 开发者手写 + AI 辅助                        │
│  二期: MCP 工具(查元数据/校验/保存) + 内置对话       │
├─────────────────────────────────────────────────┤
│  表服务(不动)                                    │
│  自定义查询语言 / 指标·维度·聚合查询 / 标准化 API    │
├─────────────────────────────────────────────────┤
│  GraphQL / ADS / DWS·DWD / ODS / 数据湖(不动)     │
└─────────────────────────────────────────────────┘
```

各层职责单一:规格只描述"是什么",运行时决定"怎么渲染",表服务决定"数据怎么算"。

**A2UI 扩展位说明:** 页面规格是唯一资产,渲染通道可以多路并存。统一运行时是当前唯一通道;当 A2UI 协议成熟(v1.0 定稿、生态稳定)后,可新增"A2UI 适配器"作为第二通道——把同一份规格翻译成 A2UI 消息流,让支持 A2UI 的 Agent 宿主直接渲染看板,无需部署我们的运行时。规格本身不感知渲染通道的存在,这正是 ADR-0001 选择自研领域 DSL、不把资产绑定协议的架构收益:接入 A2UI 是"加一个翻译器",不是"改资产格式"。

## 3. 页面规格(领域 DSL)

设计原则(ADR-0001、ADR-0003):

1. **严格声明式。** 只表达查询引用、布局、图表类型、筛选联动;禁止表达式、脚本、自定义样式。同环比、衍生计算等一律下沉到表服务的指标定义。
2. **以 BI 语义为一等公民。** 指标、维度、聚合、筛选是规格的原生概念,不是通用组件树上的挂件。
3. **可完整校验。** JSON Schema 结构校验 + 对表服务元数据的语义校验(指标/维度是否存在、聚合是否合法),两级校验为 AI 生成提供自动纠错回路。
4. **带版本。** `specVersion` 字段,运行时只支持当前大版本,升版提供迁移脚本。

### 规格结构示例

```json
{
  "specVersion": "1.0",
  "id": "sales-overview",
  "title": "销售总览",
  "filters": [
    { "id": "f-time", "type": "timeRange", "default": "last30d" },
    { "id": "f-region", "type": "dimension", "dimension": "region" }
  ],
  "layout": { "type": "grid", "columns": 12 },
  "widgets": [
    {
      "id": "w-gmv",
      "type": "metricCard",
      "position": { "x": 0, "y": 0, "w": 3, "h": 2 },
      "query": {
        "metrics": ["gmv"],
        "filters": { "subscribe": ["f-time", "f-region"] }
      }
    },
    {
      "id": "w-trend",
      "type": "lineChart",
      "position": { "x": 3, "y": 0, "w": 9, "h": 4 },
      "query": {
        "metrics": ["gmv", "order_count"],
        "dimensions": ["date"],
        "granularity": "day",
        "filters": { "subscribe": ["f-time", "f-region"] }
      }
    },
    {
      "id": "w-by-region",
      "type": "barChart",
      "position": { "x": 0, "y": 4, "w": 6, "h": 4 },
      "query": {
        "metrics": ["gmv"],
        "dimensions": ["region"],
        "filters": { "subscribe": ["f-time"] }
      },
      "interactions": [
        { "on": "click", "writeFilter": "f-region", "value": "$dimension.region" }
      ]
    }
  ]
}
```

### 数据引用:结构化查询

规格中不出现查询语句字符串。每个图表声明一个结构化查询对象(指标 + 维度 + 筛选 + 聚合粒度),运行时翻译为表服务请求。好处:逐字段校验、AI 生成可控、表服务查询语言演进不影响规格资产。

### 联动模型:页面级筛选状态

- 页面声明若干筛选器(维度筛选、时间筛选),构成共享筛选状态;
- 图表通过 `filters.subscribe` 声明订阅哪些筛选器;
- 图表交互(如柱状图点击)通过 `interactions.writeFilter` 回写筛选状态,实现联动下钻;
- 组件之间不直接连线,所有联动只通过筛选状态传递,复杂度封顶。

## 4. 统一运行时

技术栈(ADR-0002):

| 项 | 选型 | 说明 |
|---|---|---|
| 框架 | Svelte 5 + SvelteKit(static/SPA 模式) | 不用 SSR,构建产物为静态站点 |
| UI 组件 | shadcn-svelte + Tailwind CSS | 社区移植版,组件代码归我们所有 |
| 图表 | ECharts | 框架无关、图表类型全、含地图 |
| 布局 | 12 列网格 | 组件声明 x/y/w/h,与 Grafana/Rill 同款模型 |
| 校验 | JSON Schema(Ajv 或同类) | 结构校验;语义校验调表服务元数据接口 |

### 运行时职责

1. **规格加载:** 统一抽象接口 `SpecProvider`(一期实现:静态文件;二期实现:平台 API)。从第一天起运行时只依赖该接口,存储切换不改运行时。
2. **规格校验:** 加载后先过两级校验,失败给出结构化错误(供人看,也供二期 AI 修复回路用)。
3. **筛选状态管理:** 页面级 store,处理订阅、回写、URL 同步(筛选状态可分享)。
4. **查询编排:** 将各图表的结构化查询翻译成表服务请求,处理去重、并发、缓存、加载态与错误态。
5. **组件渲染:** 封闭组件集,按规格实例化。

### 组件纯渲染原则

组件只渲染"静态数据",一切"动态"归运行时:

- 组件通过 props 接收两样东西:**数据快照**(运行时查询编排的结果)和**展示配置**(来自规格);
- 组件内部**不发请求、不访问全局状态、不管理加载/错误/空态**——这些统一由运行时的查询编排层处理,组件拿到的永远是就绪的数据;
- 组件对外的唯一输出是**交互事件上抛**(如柱状图点击),由运行时决定事件如何回写筛选状态,组件不感知联动逻辑;
- 筛选器组件同样纯渲染:显示运行时传入的候选项与当前值,变更时只上抛事件,不直接写筛选状态。

收益:组件可脱离运行时独立开发和测试(mock 一份数据快照即可)、图表库可替换、组件语义清晰,未来接 A2UI 时能直接映射为 catalog 组件定义。

### 一期组件集

- **展示:** 指标卡、折线图、柱状图、饼图、表格、环形进度、地图
- **筛选:** 维度筛选器、时间范围筛选器

组件集封闭是刻意的:运行时不是通用 UI 引擎,新组件类型走平台版本发布,不走页面级扩展。

## 5. 治理与工程流程

### 一期:规格即代码(ADR-0004)

- 规格文件存 Git 仓库,目录即命名空间;
- PR 评审 = 页面发布评审;CI 跑两级校验,校验不过不能合并;
- 合并后静态部署,运行时按路由加载对应规格;
- 权限:接公司 SSO,页面级访问控制(一期可用路由级配置实现);行级/数据权限由表服务承担。

### 长期:平台化

规格入库,平台后端提供版本历史、发布状态、权限 API;建设可视化管理界面(规格列表、预览、在线编辑、发布管理)。运行时无需改动(只换 `SpecProvider` 实现)。

## 6. 三期路线图

### 一期:地基(DSL + 运行时)

**目标:开发者手写规格,能渲染出生产可用的看板页面。**

- DSL schema v1(含 JSON Schema 定义)
- 统一运行时(加载、校验、筛选状态、查询编排、9 个组件)
- 表服务查询适配层 + 元数据接口对接
- Git 工作流 + CI 校验 + 静态部署 + SSO 接入
- 用 2~3 个真实业务看板验收

**验收标准:一个新看板从需求到上线,只写规格不写代码,耗时明显低于 Angular 模式。**

### 二期:AI 生成规格

**目标:AI 通过对话生成和修改页面规格。**

- 平台暴露 MCP 工具:`查询指标/维度元数据`、`校验规格`、`保存规格`、`预览页面`
- 任意 Agent(Cursor、内部 Agent 平台等)可接入;平台内置对话界面只是其中一个客户端
- 生成回路:检索表服务元数据 → 生成规格 → 校验 → 错误回传自动修复 → 预览确认
- 平台后端 + 规格入库 + 可视化管理界面在本期落地

### 三期:AI 数据分析

**目标:从"AI 搭页面"到"AI 做分析"。**

- 对话式取数与归因分析(基于表服务的结构化查询,不生成 SQL)
- 分析结论沉淀为规格片段,一键插入看板
- 探索生成式 UI:按对话即时生成临时分析视图
- A2UI 接入评估:若协议已成熟(v1.0 定稿、生态稳定),立项"A2UI 适配器"作为第二渲染通道(见第 2 节架构图),把规格翻译为 A2UI 消息流供 Agent 宿主渲染;组件纯渲染原则保证组件集可映射为 A2UI 自定义 catalog

## 7. 一期搭建清单

按依赖顺序:

1. **DSL 定义包 `spec-schema`:** TypeScript 类型 + JSON Schema + 校验器(结构校验、语义校验接口),独立 npm 包,供运行时和 CI 共用。
2. **表服务适配层 `table-service-client`:** 结构化查询 → 表服务请求的翻译器;元数据接口(指标/维度目录)封装。
3. **运行时应用 `runtime`:** SvelteKit 工程,`SpecProvider` 抽象 + 静态文件实现,筛选状态 store,查询编排器,9 个组件(shadcn-svelte + ECharts)。
4. **规格仓库结构:** `specs/` 目录 + CI 校验流水线 + 静态部署配置。
5. **SSO 与页面级权限。**
6. **验收看板:** 挑 2~3 个存量 Angular 看板中有代表性的,用规格重写对比。
