# PRD:MetricCanvas 一期——领域 DSL 与统一运行时(地基)

> 状态:已发布为 issue #1;2026-07-19 按《组件分析.md》完成组件集二次校准
> 依据:`docs/solution.md`、`CONTEXT.md`、`docs/adr/0001~0005`、`中间层分析.md`(存量系统与表服务调研报告)、`组件分析.md`(存量看板组件使用盘点与优先级清单)

## Problem Statement

看板开发者在现有 Angular 高代码模式下,每个页面都要手写查询、布局、状态和交互逻辑:开发一个看板以周计,查询与展示耦合导致同一指标在不同页面口径漂移,页面以源代码为资产使得复用、评审、审计都很困难,AI 也无法稳定地理解和修改这些页面。业务方要一个新看板,等的是排期;开发者做一个新看板,写的是重复的胶水代码。

## Solution

一期交付"规格驱动"的看板地基:开发者不再写页面代码,而是手写一份声明式的页面规格(领域 DSL,JSON 文件),提交进 Git 仓库,经 CI 两级校验(结构 + 对元数据快照的语义校验)后静态部署;统一运行时(Svelte)加载规格,把结构化查询翻译成表服务请求,渲染出带筛选联动和跨页下钻的完整看板。开发过程有热加载和 mock 数据模式,产出的看板通过自动索引页可被发现,经公司 SSO 访问。

一个新看板从需求到上线:写一份 JSON、提一个 PR、合并即发布。指标口径由表服务统一,页面变更可 diff 可评审,规格资产为二期 AI 生成铺路。

## User Stories

**规格编写与校验**

1. 作为看板开发者,我想用 JSON 规格声明一个包含指标卡和折线图的页面,不写任何前端代码,以便一天内交付一个新看板。
2. 作为看板开发者,我想在规格中以结构化查询(指标+维度+筛选+粒度)声明数据需求,以便不用学习表服务的查询语言细节。
3. 作为看板开发者,我想在保存规格时立刻得到结构校验错误(字段拼错、类型不对、布局越界),以便在提交前修复。
4. 作为看板开发者,我想得到语义校验错误(指标不存在、维度不可用于该指标、聚合方式不合法),以便不把坏查询发到线上。
5. 作为看板开发者,我想在编辑器里获得规格的自动补全和内联校验(JSON Schema),以便凭提示写出合法规格。
6. 作为看板开发者,当我引用的指标在表服务中不存在时,我想让校验器区分"我写错了"(SCHEMA_ERROR)和"指标还没建"(METRIC_GAP),以便后者走需求通道而不是被当作 bug。
7. 作为看板开发者,我想本地一条命令跑完整校验(validate CLI),以便 pre-commit 和 CI 用同一套逻辑,不出现"本地过、CI 挂"。

**开发体验**

8. 作为看板开发者,我改动规格文件后想让浏览器秒级热刷新,以便所见即所得地调布局。
9. 作为看板开发者,在表服务不可用或没有权限的环境,我想用 mock 数据模式开发,以便离线也能全速工作。
10. 作为看板开发者,我想用 `sync-catalog` 一条命令更新元数据快照,以便表服务上了新指标后立刻可用。

**页面浏览与交互**

11. 作为业务用户,我想打开看板 URL 就看到数据(经 SSO 登录),以便不需要任何安装和配置。
12. 作为业务用户,我想通过页面顶部的维度筛选器和时间筛选器过滤所有订阅图表,以便自助分析。
13. 作为业务用户,我想点击柱状图的一根柱子让页内其它图表联动过滤(页内下钻),以便追问"华东的趋势是什么样"。
14. 作为业务用户,我想点击总览页图表跳到明细页并自动携带筛选条件(跨页下钻),以便从宏观到微观连续分析。
15. 作为业务用户,我想把带着当前筛选状态的 URL 发给同事,以便对方打开看到完全相同的视图。
16. 作为业务用户,我想在数据加载时看到骨架屏、查询失败时看到明确错误、结果为空时看到空态提示,以便不对着白屏猜。
17. 作为业务用户,我想在自动索引页浏览全部看板(标题/描述/分组),以便不依赖别人发 URL 也能找到我要的看板。

**治理与发布**

18. 作为评审者,我想在 PR 里 diff 出"这次改了哪个图表的哪个查询",以便像评审代码一样评审看板变更。
19. 作为平台维护者,我想让 CI 阻止任何校验不过的规格合并,以便线上永远不出现坏规格。
20. 作为平台维护者,我想让规格带 specVersion 并让运行时兼容 N/N-1 两个大版本,以便 DSL 演进不炸历史页面。
21. 作为数仓团队成员,我想收到带业务语境的 metric-gap issue,以便按真实页面需求排指标建设优先级。
22. 作为平台维护者,我想让页面级访问控制随规格声明并由 SSO 执行,以便权限不脱离规格资产管理。(具体载体待表服务提炼报告校准)
23. 作为存量系统维护者,我想通过 iframe/Web Component 把新看板嵌进现有 Angular 应用,以便过渡期两代系统共存。

**运行时行为(开发者视角的平台承诺)**

24. 作为看板开发者,我想让两个声明了相同生效查询的图表只触发一次表服务请求(去重),以便页面打开不放大后端压力。
25. 作为看板开发者,我想让筛选变更只重查订阅了该筛选器的图表,以便联动不引起全页重刷。
26. 作为看板开发者,我想让组件只接收就绪的数据快照(加载/错误/空态由运行时统一处理),以便组件行为完全可预测。
27. 作为平台开发者,我想让规格加载走 SpecProvider 抽象,以便二期切换平台 API 时运行时零改动。

30. 作为看板开发者,我想用文本组件放置标题、说明和带参跳转链接,以便看板不只有图表还有叙事结构。
31. 作为业务用户,我想在表格中翻页、排序,以便浏览大结果集(表服务不返回总条数,分页 UI 按"盲翻"或单独 count 设计)。

**验收**

28. 作为项目负责人,我想看到 2~3 个真实存量看板用规格重写并上线,以便验证 DSL 表达力覆盖真实需求而非演示场景。
29. 作为项目负责人,我想对比"规格路线"与"AI 直写高代码页面"的交付速度与治理增益,以便用诚实的对照组论证平台价值。

## Implementation Decisions

**模块划分**(深模块,接口小而稳):

| 模块 | 接口 | 封装 |
|---|---|---|
| `spec-schema` | `validate(spec, catalog) → TypedError[]`;TS 类型;validate CLI | 整个 DSL 定义与两级校验;错误分型 SCHEMA_ERROR / METRIC_GAP |
| `table-service-client` | `fetchData(effectiveQuery) → rows`;`getCatalog()`;`sync-catalog` CLI;mock 实现 | 结构化查询→表服务请求的翻译、返回归一化、元数据快照生成 |
| `SpecProvider` | `load(pageId) → spec` | 存储介质(一期静态文件) |
| 筛选状态 store | `subscribe / write / toURL / fromURL` | 联动唯一总线;URL 序列化 |
| 查询编排器 | `orchestrate(widgets, filterState) → snapshot 流` | 生效查询合成、去重、缓存、并发、加载/错误/空态 |
| 纯渲染组件集(9) | props 进(数据快照+展示配置)、事件出 | ECharts/shadcn-svelte 实现细节 |
| 应用壳 | 路由 → 页面 | 组装、SSO、页面级权限、索引页 |

**关键决策**(详见 ADR 与 solution.md):

- 自研领域 DSL,严格声明式,禁表达式/脚本/自定义样式(ADR-0001/0003);自研运行时而非开源 BI(ADR-0005);Svelte + shadcn-svelte + ECharts + SvelteKit 静态 SPA(ADR-0002);一期规格存 Git(ADR-0004)。
- 规格核心概念:结构化查询、页面级筛选状态(页内联动)、`navigate` 跨页下钻(URL 载体)、12 列网格布局、specVersion(运行时兼容 N/N-1,CI 对 N-1 警告)。
- 数据流:生效查询 = 结构化查询 × 订阅筛选器当前值;组件只消费数据快照;编排器是唯一有状态调度方,适配层无状态可复用于 CI/二期 MCP。
- 元数据快照:入 Git 的供给侧清单,CI 语义校验的确定性参照;零数据行,业务数据永远实时查询、不落盘。
- 第 0 号任务:确认表服务元数据接口形态(完备/残缺/缺失三情形三对策),前置于一切开发。
- 组件集(已按《组件分析.md》消费侧 23 页面真实使用频次二次校准):指标卡、折线、柱状、饼图、表格、地图、文本 + 维度筛选器、时间筛选器(共 9 个)。表格是存量使用频率最高、交互最重的组件,为一期复杂度重点;其分页、表头排序、表头筛选(下拉多选 + 日期)是 P0 表格子能力,须在 DSL 中显式建模。维度筛选器须支持下拉多选、树形选择、搜索、tab 四种展示形态——tab 形态即存量高频 ti-tabs 的维度/区域切换场景(8 个页面),在本平台语义下是"写入筛选状态的筛选器",不是独立容器组件;时间筛选器支持日期范围与日期时间。环形进度从一期组件集降级至 P1:存量实为行内进度条与圆形加载,非环形进度形态,一期可用指标卡 + 百分比过渡。
- 图表展示配置面(来自《组件分析.md》§2 真实配置扫描,即 DSL 展示配置的字段范围):折线须支持平滑、面积渐变、堆叠、双轴;柱状须支持堆叠、圆角、横向、双轴(横向柱状同时覆盖存量两个自定义水平条形组件);饼图须支持环形 radius 与引导线开关;指标卡须支持 unit/prefix、千分位格式化、跳转链接。
- P1 清单(一期可缺、尽早补,验收看板选型若命中则前置):进度条(行内/圆形)、迷你趋势图(折线图精简模式,表格行内 sparkline)、弹窗钻取容器(行内缩略图/趋势弹窗,存量 5 页面)、图表/表格切换;表格进阶能力:多级表头(colspan/rowspan)、动态行合并、声明式单元格类型(进度条/迷你趋势/链接)、全局汇总行、列宽可调。

**表服务对接事实**(已由调研报告《中间层分析.md》校准):

- **协议**:GraphQL-over-HTTP POST 到单一端点,查询体包装为 `{"apiQuery":"...","isTest":...}`;非标准 GraphQL,标准客户端(Apollo 等)不可直接用,适配层自写 fetcher。
- **查询方言**:6 个自定义指令(`@limit`/`@offset`/`@where`/`@order`/`@function`/`@connect`)。结构化查询的翻译映射:指标 → 字段 + `@function` 聚合(或内置 `{col}_sum` 保留字段);筛选 → 生成 `@where` 类 SQL 条件字符串(适配层维护操作符白名单,防注入、防後端解析失败);排序 → `@order(type, priority)`;分页 → `@limit`+`@offset`。
- **时间粒度**:表服务在服务定义时固定粒度(年/月/周/日/时/分/秒),查询时不可动态切换。DSL 的 `granularity` 在适配层映射为对预定义字段/服务的引用;粒度映射表进元数据快照。
- **JOIN**:服务定义时静态配置,不支持查询时动态 JOIN;DSL 不暴露 JOIN 概念(本就符合严格声明式边界)。
- **同环比**:表服务无内置,靠 SQL `case when` 在服务定义层实现——与 ADR-0003"复杂计算下沉表服务"一致,但需向业务明确:新增同环比指标要走数据服务定义流程。
- **批量与健壮性**:批量查询上限 5(查询编排器分批);响应不返回总条数(表格分页策略:盲翻或单独 count);表服务无超时/重试约定,适配层自实现(超时 30s、幂等查询指数退避重试)。
- **响应信封**:`retCode: "CBC.0000"` 为成功,`data` 以服务码为 key;错误码清单见报告 §3.3。
- **鉴权**:无 Bearer Token。宿主门户注入 `window.currentLoginUser` + Session Cookie,请求头需携带 `x-operator-id`/`tenantId`/`appId`/`cftk`;未登录信号触发 `window.relogin()`。数据行级权限由后端"控制字段"下推,前端零感知——只需带对请求头。
- **元数据现状**(第 0 号任务的答案):指标目录、维度目录**完全缺失**;表目录部分具备(`services/list` + GraphQL `__type` 内省)。后端已有 `MetricBaseInfo` resolver(返回 metric_code/metric_name_zh/scope),前端未封装。**第 0 号任务走"残缺→共建"路线**:快照初版由 `services/list` + `__type` 内省 + `MetricBaseInfo` 拼装,同时与后端确认扩展"可用维度/可用聚合"字段(报告估算:前端封装 1-2 人日,后端扩展 3-5 人日)。
- **组件集校准**:《中间层分析.md》§5.2 的粗排已被《组件分析.md》(消费侧 23 页面逐一扫描)取代,后者为组件优先级的权威依据。关键修正:Tab 实为高频(8 页面,以维度筛选器 tab 形态覆盖,非独立组件);仪表盘/轮播消费侧零使用,一期明确不支持;**地图并非纯新需求**——消费侧 3 个页面在用 dv-map(world/region 底图 + 区域着色 + 散点叠加);文本组件保留(标题/说明,高频刚需,成本极低);表格明确为一期复杂度重点。
- **验收看板候选**(报告 §5.5):`lowcode-example`(能力最全)、`lowcode-use-example`(用户视角)、`preview` 渲染的真实 IOC 业务页面。

**仍待确认**(不阻塞动工,见报告附录 B):

1. `isTest` 字段的精确含义(适配层需要知道生产该传什么);
2. `@function` 支持的完整聚合函数清单、`@where` 支持的完整操作符(决定白名单范围);
3. `MetricBaseInfo` resolver 的完整字段与扩展成本(第 0 号任务首个行动);
4. 部署形态:独立站点为既定决策,但若需进现有门户,宿主用 qiankun/single-spa 微前端(`window.currentLoginUser` 等全局契约),iframe/Web Component 嵌入与 qiankun 接入二选一需届时评估。

## Testing Decisions

- **好测试的标准**:只测公开接口的外部行为,不测实现细节;测试读起来像规格说明("订阅了 f-region 的图表在筛选变更后收到新快照"),重构内部实现不应破坏测试;期望值来自独立事实(手工算例、规格文档),不得复算实现逻辑。
- **测试的模块(四个,均为纯逻辑、seam 清晰)**:
  1. `spec-schema` 校验器——合法/非法规格样例集,断言错误类型与定位;
  2. `table-service-client` 翻译器——结构化查询→请求的映射表驱动测试(mock 表服务 HTTP 边界);
  3. 筛选状态 store——订阅、回写、URL 序列化往返;
  4. 查询编排器——生效查询合成、去重、"筛选变更只重查订阅者"(mock 适配层接口)。
- **不测**:SpecProvider(接口太薄)、纯渲染组件(mock 快照做开发期目验)、应用壳(验收看板兜底)。
- **先例**:绿地仓库无先例;约定 Vitest,测试与被测包同置。

## Out of Scope

- AI 生成/修改规格、MCP 工具、平台内置对话(二期);
- 平台后端、规格入库、可视化管理界面、在线编辑(二期);
- 对话式取数、归因分析、生成式 UI(三期);
- A2UI 适配器(未立项,见 solution.md 附录);
- 行级/数据权限(表服务承担);
- 存量 Angular 页面迁移(明确不做);
- 长尾图表:独立散点、漏斗、桑基、雷达、热力、树图、关系图、仪表盘 gauge、日历、水球——存量消费侧零使用(《组件分析.md》§3.2),一期明确不支持;
- 表格长尾能力:CSV 导出、虚拟滚动、详情表展开、列显隐齿轮、单元格条件格式——存量消费侧零使用(《组件分析.md》§4.1),一期明确不支持;
- 组件自动化测试、像素级样式定制、规格内表达式(ADR-0003 明确禁止)。

## Further Notes

- 全链单点风险:元数据接口(第 0 号任务)。若表服务无目录接口且共建周期长,快照转人工维护,语义校验价值不减但维护成本上移。
- DSL 表达力缺口的处理程序:记录缺口 → 三选一(进 DSL 下版本 / 下沉表服务 / 明确拒绝),禁止规格内开逃生舱。
- 一期结束时的口碑风险:开发者对受限 DSL 的评价决定二期政治空间,DX 三件套(热加载/mock/validate CLI)是对冲手段,优先级不可降。
